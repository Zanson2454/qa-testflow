# PRD-001 需求确认（v1）

> 说明：本文档是 `PRD -> 测试用例 -> E2E验收` 的第一步确认产物，只定义“要验证什么”，不包含自动化实现细节。

## 1. 需求目标

- 在 `console` 支持配置“公开页面”。
- 保存配置后自动生成访问链接，支持复制。
- 用户通过该链接访问 `portal` 时可无需登录。
- 方案目标是解决“公开页面可打开但数据因认证/鉴权受阻不可用”的问题。

## 2. 范围（in_scope）

- `console` 侧公开页面配置流程：
  - 选择页面
  - 保存配置
  - 获取并复制公开链接
- `portal` 侧访问流程：
  - 未登录访问公开链接
  - 正常场景下页面可见且关键数据可加载
  - 非公开场景下访问被拦截（无权限/不可访问）

## 3. 非范围（out_of_scope）

- 白名单后台细粒度配置 UI 的自动化（若由其他系统维护）。
- 复杂压测、性能压测、跨端兼容矩阵（后续迭代）。
- 首轮不覆盖接口白盒逻辑验证，仅做黑盒端到端验收。

## 4. 关键业务口径

- 公开页面“真正可用”定义为：
  - 页面框架可访问；
  - 页面关键业务数据可成功加载；
  - 不出现认证/鉴权导致的关键功能阻断。
- 未公开页面链接访问应被系统拦截并给出可识别提示。

## 5. 已知约束

- 登录策略：`console` 自动化先支持 Cookie 方案。
- 页面定位策略：优先 `data-testid`，不足时在调试阶段补充稳健定位。
- 执行方式：按 harness 流程推进，先需求确认、再工程实现。

## 6. 风险与待确认

- 风险1：`data-testid` 覆盖不足可能导致定位波动。
- 风险2：不同环境权限模型差异可能影响未登录拦截行为。
- 风险3：关键数据接口返回信号可能存在异步时序抖动。
- 待确认1：配置关闭公开后的历史链接是否必须立即失效（产品规则）。

## 7. 需求确认结论

- 本 PRD 可进入“用例映射与验收标准定义”阶段。
- 通过本确认后，后续脚手架与自动化实现必须严格对齐本文定义的范围与口径。
